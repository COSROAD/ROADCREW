· **"그래서 아까 확인 하라고 한겁니다. 이런일이 생길까봐서."**

- **push 전**: `</html>` 끝 · script 짝 · node --check(classic/module) · 셀렉터 중복(@media 밖) ·
  없는 CSS 변수 · 끊어진 호출 · 함수/전역 중복 정의 · 버전==version.json

- **동작 검증**: 문법만으로 부족하면 **가짜 DOM/Firebase 로 실제 실행**해 화면을 그려본다.
  ⚠️ 변수 값만 보지 말 것 — **화면이 몇 번 그려지는지**를 봐야 깜빡임·덮어쓰기가 잡힌다.

> 프리존 님 승인: "무의미한 반복적인 실수는 하지 않아야 됩니다."
> 이 원칙은 모든 클로드(코워크·Code 탭)에 공통 적용된다.

## 원칙 A. 외부 API 코드는 대조 후 작성
외부 API(티맵·카카오·CODEF·알리고 등)를 부르는 코드는
공식 문서·샘플과 **항목 단위 대조표**를 만든 뒤 작성한다.
주소(URL)·필수 파라미터·값 형식(번호 시작값, 자릿수, 인코딩)을 하나씩 대본다.
대조 없이 쓴 호출 코드는 커밋하지 않는다.
(사례: 문서에 없는 옛 주소 optimized-order 사용, viaPointId 0부터 시작 → 9401)

## 원칙 B. 저장 코드는 실제 저장값을 눈으로 확인
데이터를 저장하는 코드는 가짜 실행(스텁 테스트)으로
**실제 저장되는 값**을 출력해 확인한 결과를 남긴다. "될 것이다"는 인정 안 됨.
기본값 낙하(fallback)가 있으면 반드시 "언제 낙하하는지"를 적는다.
(사례: 엑셀 업로드가 전 회원에게 기관 좌표 복사 — 4업종 공통)

## 원칙 C. 오류 수정은 오류 원문 확보가 먼저
오류를 고칠 때는 **오류 원문(서버 로그·응답 본문)**을 확보한 뒤 수정한다.
원문 없이 추측 수정 커밋 금지. 로그가 없으면 로그를 심는 것이 1차 수정이다.
(사례: 거절 사유 로그를 4차 수정에서야 추가 — 1차에 했으면 수정 5회가 2회로 줄었음)
